# Как корпус проекта работает для журналистики, data-журналистики и OSINT

Журналистика, data-журналистика и OSINT держатся на одних и тех же четырёх опорах: **провенанс, верификация, аудит, защита источника**. Этот репозиторий три из них уже закрывает.

## Главная идея

Git-репозиторий с дисциплиной «сырьё → черновик → ревью → публикация» — это машина воспроизводимости и chain of custody. То, что для редакции «как мы вели статью», для расследования — «как мы можем доказать каждое слово».

## Карта: что в репо → что в расследовании

| Компонент проекта                 | Функция в расследовании                                                          |
| --------------------------------- | -------------------------------------------------------------------------------- |
| `sources/` + `MANIFEST.md` (хеши) | Сбор улик + **chain of custody**: доказать, что источник не подменён             |
| `git log`                         | **Неизменяемый аудит-след**: кто, что, когда утверждал; как факт эволюционировал |
| Ветка на материал                 | Изолированное досье по делу                                                      |
| `draft → review → published`      | Гейт фактчекинга: ничего не выходит без прохода через ревью                      |
| `/review`                         | Автопроверка (сейчас редполитика — расширяется в чек-лист фактчека)              |
| `ref/`                            | Библиотека паттернов/эталонов                                                    |
| `sources/archive/`                | Сохранённые версии фактуры — источник восстановим даже после правок              |
| `session-notes` + память          | Институциональная память между расследованиями                                   |
| `.opencode/` агенты               | LLM-ассистент на сборе/верификации/анализе/черновике                             |

## Как ложится на цикл расследования

OSINT-цикл (**Сбор → Верификация → Анализ → Документирование**) ложится один в один:

1. **Сбор** → `sources/<дело>/`, при добавлении — `manifest.sh` фиксирует хеш в момент получения. Веб-страница, скрин, документ, дамп — всё хешируется на входе.
2. **Верификация** → перекрёстная сверка нескольких источников (агент находит противоречия), целостность подтверждается `shasum`. Источник изменился/удалён в сети — твоя хешированная копия доказывает, что там было.
3. **Анализ** → агент по корпусу источников: извлечение сущностей, таймлайн, связи, аномалии.
4. **Документирование** → `articles/` + `notes`, каждый коммит датирован = таймлайн «когда что узнали».

## По трём дисциплинам — конкретно

### Журналистика (расследовательская)

- Дело = ветка. Источники с `README.md`: от кого, когда, гриф.
- Когда героя материала юристы спросят «вы это выдумали» — `git log --follow` + хеш в манифесте показывают точную версию источника, из которой взята фраза, с датой.
- **Защита источника:** конфиденциальное и PII не в git, наружу — только хеши. Источник не утечёт через репозиторий.

### Data-журналистика

- Сырые данные (`*.xlsx`, `*.csv`) лежат в `sources/`. Пайплайн обработки — в `scripts/`, выход — производный датасет, оба (сырой + производный) хешируются в манифесте.
- «Show your work»: коммит статьи ссылается на версию данных → любой воспроизводит цифру, прогнав скрипт на хешированном входе.
- `git diff` между черновиками ловит, как поменялось число — защита от ошибок в расчётах.

### OSINT

- **Опсек = минимальный след:** репо локальный (нет remote), секреты и NDA вне git — ровно то, что нужно расследованию.
- **Доказательность во времени:** время коммита = «когда это было у нас». Таймлайн дела реконструируется из истории.
- **Корпус как база знаний:** все собранные материалы ищутся и кросс-связываются агентом по сущностям/датам.

## Роль LLM-слоя — и его дисциплина

Агенты ускоряют каждый этап: расшифровка интервью, суммаризация дампов, поиск противоречий, извлечение сущностей, черновик. Но — **модель предлагает, источник и человек подтверждают**. В расследовании это не стилистика, а правило выживания: каждое утверждение трассируется к хешированному источнику, не к «модель так сказала».

## Честные пробелы (что добавить для серьёзного применения)

| Пробел                             | Что добавить                                                                          |
| ---------------------------------- | ------------------------------------------------------------------------------------- |
| Нет связи «утверждение ↔ источник» | Инлайн-ссылки на источник в тексте или «реестр утверждений» (claim → файл+строка+хеш) |
| Верификация не принудительна       | Гейт фактчек-чек-листа перед `published` (расширить `/review`)                        |
| Веб-источники                      | Скрипт capture: архив HTML + скрин + хеш в момент захвата                             |
| Время — только git-таймстамп       | Для adversarial-дел — криптографический таймстамп (RFC 3161)                          |
| Один следователь                   | Контроль доступа к хранилищу источников (сейчас локально, один пользователь)          |

## Вывод

Сильные стороны для расследований уже стоят: **провенанс, аудит-след, защита источника, воспроизводимость**. Это редкое сочетание — большинство журналистов ведут это в разрозненных папках без целостности и истории. Фронтир, который стоит достраивать, — **принудительная верификация** (трассировка утверждений к источникам) и **захват веб-улик**. Оба — небольшие надстройки поверх того, что уже есть.
